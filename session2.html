Understanding Software Development Life Cycle (SDLC)

SDLC is a framework that defines the process used by organizations to build software applications from inception to retirement. It provides a structured approach to software development.

Why Do We Need SDLC?

  • Structure: Provides a systematic approach to development
  • Quality: Ensures high-quality software delivery
  • Cost Control: Helps manage budget and resources
  • Risk Management: Identifies and mitigates risks early
  • Communication: Creates common understanding among stakeholders

Common SDLC Phases (Universal to Most Models)

1
Planning: Define project scope, objectives, and feasibility
2
Requirements Analysis: Gather and document what the software must do
3
Design: Create architecture and detailed design specifications
4
Implementation/Development: Write the actual code
5
Testing: Verify the software works correctly
6
Deployment: Release software to users
7
Maintenance: Fix bugs, add features, provide support

Part 1: Waterfall Model (90 minutes)

What is the Waterfall Model?

The Waterfall Model is a sequential, linear approach to software development where each phase must be completed before the next phase begins. Like a waterfall, progress flows steadily downward through phases.

Waterfall Phases (Sequential Flow)

1
Requirements Gathering & Analysis
Collect all requirements upfront. Create detailed specification documents.

2
System Design
Design system architecture, database schema, UI mockups.
‚Üì
3
Implementation (Coding)
Developers write code based on design documents.
‚Üì
4
Testing
QA team tests the complete system for bugs and issues.
‚Üì
5
Deployment
Release the software to production environment.
‚Üì
6
Maintenance
Fix bugs, provide updates and support.

Key Characteristics of Waterfall

  • Sequential: One phase at a time, no overlap
  • Documentation-Heavy: Extensive documentation at each phase
  • Rigid: Difficult to change requirements once started
  • Phase Gates: Must complete and approve each phase before proceeding
  • Big Bang Deployment: Everything released at once

üìã Practical Example: Building a Banking System (Waterfall)

Scenario: A bank wants a new customer management system.

Month 1-2: Requirements

  • Business analysts meet with bank executives
  • Document every feature: account creation, deposits, withdrawals, reports
  • Create 200-page requirements document
  • Get sign-off from all stakeholders

Month 3-4: Design

  • Architects design database schema (20+ tables)
  • Create UI mockups for every screen
  • Design system architecture diagrams
  • No coding yet!

Month 5-10: Implementation

  • Developers write code for all features
  • Front-end team builds all UI screens
  • Back-end team implements all APIs
  • Database team sets up tables and stored procedures

Month 11-12: Testing

  • QA team receives complete system
  • Tests all features end-to-end
  • Finds 200+ bugs (too late to fix major design flaws!)

Month 13: Deployment

  • Deploy entire system to production
  • Bank staff see the system for the first time
  • "This is not what we wanted!" (Requirements changed in 13 months)

Total Time: 13 months

Problem: By the time it's deployed, business needs have changed!

‚úÖ Advantages of Waterfall

  • Simple and easy to understand
  • Well-documented process
  • Works well for small, clear projects
  • Easy to manage (clear milestones)
  • Good for fixed-price contracts
  • Suitable for projects with stable requirements

‚ùå Disadvantages of Waterfall

  • No working software until late in the cycle
  • Cannot accommodate changing requirements
  • High risk and uncertainty
  • Customer sees product only at the end
  • Testing happens too late
  • Difficult to estimate time and cost accurately

When to Use Waterfall?

Good For:

  • Projects with clear, unchanging requirements (e.g., government contracts)
  • Small projects with well-understood technology
  • Projects where documentation is critical (medical devices, aerospace)
  • Teams geographically distributed with limited communication

Not Good For:

  • Complex, long-term projects
  • Projects where requirements may change
  • Innovative products where customer feedback is needed
  • Competitive markets requiring quick releases

Part 2: Agile Model (90 minutes)

What is Agile?

Agile is an iterative and incremental approach to software development that emphasizes flexibility, collaboration, customer feedback, and rapid delivery of working software.

The Agile Manifesto (2001)

Four core values:

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

Agile Principles (12 Key Principles)

  1. Our highest priority is satisfying the customer through early and continuous delivery
  2. Welcome changing requirements, even late in development
  3. Deliver working software frequently (weeks rather than months)
  4. Business people and developers must work together daily
  5. Build projects around motivated individuals
  6. Face-to-face conversation is the best form of communication
  7. Working software is the primary measure of progress
  8. Sustainable development (maintain constant pace)
  9. Continuous attention to technical excellence
  10. Simplicity - maximize the amount of work not done
  11. Self-organizing teams
  12. Regular reflection and adjustment

üìã Practical Example: Building an E-commerce Website (Agile)

Scenario: Startup wants to build an online shopping platform.

Sprint 1 (Weeks 1-2): MVP - Basic Product Listing

  • Day 1-2: Plan features: Show 10 products with images and prices
  • Day 3-8: Developers build product listing page
  • Day 9-10: Test and deploy to staging
  • Result: Working website with product catalog (deployed!)
  • Customer feedback: "Great! But we need a search feature"

Sprint 2 (Weeks 3-4): Add Search & Shopping Cart

  • Adapt based on feedback
  • Build search functionality
  • Add basic shopping cart
  • Result: Users can now search and add items to cart
  • Customer feedback: "Perfect! Now add payment"

Sprint 3 (Weeks 5-6): Payment Integration

  • Integrate payment gateway (Stripe/PayPal)
  • Add checkout process
  • Result: Fully functional e-commerce site!
  • Already making sales while adding more features

Sprint 4 (Weeks 7-8): User Accounts & Order History

  • Add user registration/login
  • Order history and tracking
  • Keep improving based on real user data

Total Time to First Release: 2 weeks!

Benefit: Early customer feedback, revenue from week 6, continuous improvement

üîß Practical Demo: Agile User Story

Format: As a [type of user], I want [goal] so that [benefit]

Example User Stories for E-commerce:

  • As a customer, I want to search for products by name so that I can quickly find what I need
  • As a customer, I want to save items to wishlist so that I can purchase them later
  • As a seller, I want to view sales analytics so that I can track my revenue

Popular Agile Frameworks

Scrum

  • Most popular Agile framework
  • Fixed-length sprints (1-4 weeks)
  • Roles: Product Owner, Scrum Master, Dev Team
  • Ceremonies: Sprint Planning, Daily Standup, Review, Retrospective

Kanban

  • Visual workflow management
  • Continuous flow (no fixed sprints)
  • Work-in-progress (WIP) limits
  • Board: To Do ‚Üí In Progress ‚Üí Done

‚úÖ Advantages of Agile

  • Early and frequent delivery of working software
  • Welcomes changing requirements
  • Continuous customer involvement and feedback
  • Faster detection and resolution of issues
  • Higher customer satisfaction
  • Better team morale and collaboration
  • Reduced risk (fail fast, learn fast)

‚ùå Challenges of Agile

  • Requires active customer involvement
  • Less predictable (hard to estimate final cost/time)
  • Can be chaotic without discipline
  • Minimal documentation (can be a problem later)
  • Difficult for distributed teams
  • Requires experienced team members

Part 4: DevOps Integration with Agile (45 minutes)

How DevOps Enhances Agile

While Agile focuses on faster development cycles, DevOps extends this to deployment and operations. They are complementary practices!

The Relationship:

  • Agile = Fast development iterations
  • DevOps = Fast development + Fast deployment + Continuous monitoring
  • Together = Complete pipeline from code to customer

What Agile Doesn't Address (DevOps Fills the Gap)

Gap in Agile How DevOps Addresses It
Deployment is still manual and slow Automated CI/CD pipelines
Dev and Ops still work separately Unified DevOps team
No focus on infrastructure automation Infrastructure as Code (IaC)
Limited production monitoring Continuous monitoring and feedback
Testing may still be delayed Continuous testing in pipeline
